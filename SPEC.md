# SPEC.md: 3DCG描画のしくみ 超丁寧・複数ページ版 開発仕様書（v2）

## 0. v1からの変更点（このリライトの目的）

v1は全7章を1ページに詰め込んだため、1つ1つの概念の説明が数行で終わってしまい、
初学者には「単語は出てきたが、なぜそうなるのか分からない」ページになっていた。

v2では以下の方針に全面的に切り替える。

1. **1ページ=1テーマの複数ページ構成にする。** 総ページ数が多くなることは問題にしない。
   1ページあたりの情報量を絞り、そのぶん1つの概念を何段落でも使って説明する。
2. **すべての概念を「説明テンプレート」に沿って書く。** どの概念も必ず
   「①なぜこの話をするのか → ②身近な例え → ③仕組みの説明（段階分割） → ④触って確認（インタラクティブ） → ⑤よくある誤解・つまずき → ⑥このページのまとめ」
   の6ブロックで構成する。読者は各ページで同じリズムで学べる。
3. **専門用語は初出時に必ず「ふりがな＋一言定義＋詳細説明」をセットにする。**
   用語だけ出して先に進むことを禁止する。ページ下部に用語集も置く。
4. **1つのインタラクティブ機能に対して、操作手順を必ず文章で誘導する。**
   「スライダーを動かしてみましょう」ではなく「まずFOVスライダーを50°から100°までゆっくり上げてください。
   このとき右画面のオレンジの枠が横に開くのを見てください。次に…」のように、
   何を見るべきかを1ステップずつ指示する（ガイド付きツアー）。
5. **各ページ末尾に理解度チェック（3問程度の選択式クイズ）を置く。**
   その場で正誤とやさしい解説を表示する。

対象読者・数式排除・Three.jsによるエミュレーション方針はv1と同じ。

### 3.1 ch1-coordinates.html「座標とは何か」

**このページで分かること:** 3Dの「座標」は中学のグラフとほぼ同じ／Zは奥行きの1本を足しただけ／マイナスの意味。

本文構成:

1. **なぜ座標の話から始めるのか** — 3DCGのすべてのデータは最終的に「数字3つの組」に行き着く。ここが曖昧だと後の全部が曖昧になる、という動機付け。
2. **中学のグラフを思い出す（X, Y）** — 横X・縦Y、原点、プラスマイナスの向き。教室の座席表・地図の「東に3、北に2」の例え。2Dキャンバスで点を打つミニデモ（後述デモA）。
3. **奥行きZを足す** — 「画面の手前と奥」という3本目の軸。例え:「教室の座席表(X,Y)に『何階か(Z)』を足すと校舎全体の住所になる」。右手系/左手系の混乱は出さず、本サイトでは「Zプラスが手前」と最初に1回だけ宣言して固定する。
4. **マイナスの座標が意味するもの** — 原点より左/下/奥。マイナス=悪いことではなく「向きが逆」なだけ。
5. **座標は「住所」であって「絵」ではない** — 点そのものには色も大きさもない。位置情報だけ。
6. **よくある誤解** — 「3Dソフトの画面に見えている物体がデータ本体だ」という誤解を正す。見えているのは住所リストを毎フレーム描き直した結果にすぎない。

デモA（2D→3D座標体験）:
* 左: X,Yの2つのスライダーと2Dグリッド。点が動く。
* 「奥行きを追加」ボタンでキャンバスがThree.jsの3Dビューにモーフィングし、Zスライダーが出現。
* ガイド付きツアー: X→Y→Zの順に自動で動かし、それぞれ「どの向きに動いたか」を吹き出しで確認させる。
* チェックリスト: 「Xを増やすと点が右に動くのを見た」「Zを増やすと点が手前に来るのを見た」等。

クイズ例: 「点(−2, 0, 3) の −2 が意味するのは？」等3問。

### 3.2 ch1b-vertex-polygon.html「頂点とポリゴン」

**このページで分かること:** 頂点=座標を持つ点／3つ結ぶと三角形（ポリゴン）／3Dモデル=三角形の大量の集まり。

本文構成:

1. **点に名前を付ける: 「頂点（ちょうてん／Vertex）」** — 用語導入。座標を持つ点のこと。
2. **なぜ「3点」なのか** — 2点では線にしかならない。3点あると初めて「面」が張れる。4点だと同一平面に乗らない場合がある（紙を対角にひねる例え）ので、GPUは必ず三角形で処理する。ここは初学者が最初に「なるほど」と言う定番ポイントなので厚めに書く。
3. **三角形を張る＝「点と点を結ぶ順番のリスト」を作ること** — モデルデータの正体は「頂点の住所リスト」＋「どの3点を組にするかのリスト」の2つだけ、と明言する。
4. **三角形をたくさん集めると「形」になる** — 三角形2枚で四角、12枚で立方体、数千枚で人の顔。粗いモデル/細かいモデルの違い=三角形の枚数の違い（ここが後の第4〜6章の伏線になることを予告）。
5. **よくある誤解** — 「滑らかな曲面はどこかに曲線データがある」→ ない。全部三角形。近くで見ると多面体である実例をデモで見せる。

デモB（三角形組み立てキャンバス、v1のデモを拡張）:
* 頂点3つのX,Y,Z入力欄＋3D表示（v1と同じ）に加えて:
* 「頂点を増やす」ボタンで4点目以降を追加し、三角形を2枚、3枚と張れる。
* プリセットボタン「立方体を見る」「球を見る」: 出来合いモデルをワイヤーフレーム表示し、スライダーで分割数（三角形の枚数）を変えられる。枚数カウンターを常時表示。
* ガイド付きツアーあり。

### 3.3 ch2a-local-world.html「ローカル空間とワールド空間」

**このページで分かること:** モデルは自分専用の原点で作られる（ローカル）／世界に配置する時に共通の原点に乗せ替える（ワールド）／Transformの正体。

本文構成:

1. **問題提起: モデラーが作った「木」はどこにある？** — DCCツールで作った時、木は「木ファイルの中の原点」を中心に作られている。ゲームの世界のどこに置くかは、まだ決まっていない。
2. **ローカル空間＝「そのモデル専用のものさし」** — 例え: 家具の組み立て説明書。説明書の中の座標は「その家具の中だけで通じる住所」。部屋のどこに置くかは説明書には書いていない。
3. **ワールド空間＝「世界共通のものさし」** — ゲームのステージ全体で共通の原点。例え: 部屋の間取り図。
4. **乗せ替え作業の3点セット: 位置・回転・大きさ** — UnityやUEのインスペクタで毎日触っている「Transform」がまさにこれ。移動(Position)・回転(Rotation)・拡大縮小(Scale)の3つを指定すれば、ローカルの全頂点をワールドの住所に一括変換できる。
5. **「ワールド行列」という言葉だけ覚える** — この3点セットをGPUが一瞬で適用できる形にまとめた「変換レシピカード」を行列（ワールド行列）と呼ぶ。中身の数字は見なくてよい。「レシピカードを頂点全員に配って一斉に引っ越しさせる」イメージ図を載せる。
6. **同じモデルを100本置くとき** — 頂点データは1セットのまま、レシピカード（行列）を100枚用意するだけ。だから同じ木を大量に置いても頂点データは増えない（インスタンシングの入り口。用語は脚注扱い）。
7. **よくある誤解** — 「Scaleを2にするとデータの座標が書き換わる」→ 元データは不変。変換は毎フレーム計算で適用されるだけ。だからいつでも1に戻せる。

デモC（v1の木と岩デモを拡張）:
* v1同様の2画面（プレイヤー視点/神の視点）＋木・岩のTransformスライダー。
* 追加: 「ローカル空間をのぞく」トグル。ONにすると選択モデルだけが原点にある状態の小窓が出て、ワールドへの矢印アニメーションが表示される。
* 追加: 頂点1個をクリックすると「ローカル座標→ワールド座標」の数値変化が吹き出しで表示される。
* ガイド付きツアーあり。

### 3.4 ch2b-view.html「ビュー空間 — カメラが世界の中心になる」

**このページで分かること:** カメラにはフィルムがある訳ではなく、世界の方を動かしている／ビュー空間=カメラを原点にした座標への再計算。

本文構成:

1. **問題提起: 「カメラに映る」とはどういう計算か** — 現実のカメラはレンズに光が入る。CGのカメラは何も受け取らない。では何をしている？
2. **発想の転換: カメラを動かすのではなく、世界を逆向きに動かす** — 例え: 電車の窓。自分が前に進むと、景色は後ろに流れる。GPUは毎フレーム「世界の全頂点を、カメラから見た位置に引き寄せ直す」計算をしている。この節がこのページの心臓部。図解を3枚以上使い、カメラが3歩前進する＝全物体が3歩手前に来る、を段階的に見せる。
3. **ビュー空間＝「カメラが原点(0,0,0)のものさし」** — 木の住所が「ワールドで(4, 0, −3)」から「カメラから見て前方9m・右4m」に翻訳される。
4. **これも「レシピカード」1枚（ビュー行列）** — ワールド行列と同じ仕組み。カメラの位置と向きから作られる。
5. **毎フレーム、全頂点に対して行われている** — 60fpsなら1秒に60回、何百万頂点を翻訳し直している。ここで「頂点数が多いと重い」の実感を初めて数字で持たせる（第4章への伏線）。
6. **よくある誤解** — 「カメラを増やすと世界のコピーが増える」→ 増えない。レシピカードが増えるだけ。

デモD（v1のカメラ移動デモを拡張）:
* v1同様の2画面＋カメラ移動（十字キー/ボタン）。
* 追加: 「世界を動かすモード」トグル。ONにすると左画面は固定のまま、右画面（神の視点）で**世界全体がカメラと逆向きに滑る**様子を表示。「カメラが進む」と「世界が引き寄せられる」が同じ結果になることを目で確認できる。本ページの核となるギミック。
* 数値readout（木・岩のビュー空間座標）はv1から継承し、「前方○m・右○m」の日本語表記を併記。
* ガイド付きツアーあり。

### 3.5 ch2c-projection.html「投影とクリップ — 映る範囲を決める」

**このページで分かること:** 視錐台=カメラに映る領域の箱／FOV・Near・Farの意味／遠近感（遠くのものが小さい）が作られる場所。

本文構成:

1. **問題提起: カメラは全部を映せない** — 視野の外・後ろ・遠すぎるものは映らない。どこで切っている？
2. **視錐台（しすいだい／Frustum）＝「映る範囲の箱」** — ピラミッドの頭を切った形。例え: 懐中電灯の光の広がり。4つの側面＋Near面＋Far面の6枚の壁。
3. **FOV（画角）** — 箱の開き具合。広角レンズ/望遠レンズの写真の例を添える。広げると多く映るが1つ1つは小さくなる。
4. **Near面とFar面** — なぜ手前も切るのか（カメラにめり込んだ物を描かないため）、なぜ遠くを切るのか（無限遠まで計算できないため）。ゲームで遠景が消える「ドロー距離」の正体。
5. **遠近感はここで生まれる** — 同じ大きさの物でも、箱の奥にあるほど画面上では小さく潰される（透視投影）。例え: 台形の箱を正面から押し潰して直方体にすると、奥の物ほど縮む、の図解アニメーション。
6. **クリップ＝箱の外の頂点を捨てること** — 「クリップ空間」という名前はこの切り捨て（クリッピング）から。捨てるのは描画の前。無駄な計算をしない工夫の第一歩（カリングという言葉をここで初出、第5章への伏線と明言）。
7. **よくある誤解** — 「Farを伸ばしても見た目が変わらなければタダ」→ 映る物が増えれば計算も増える。

デモE（v1の視錐台デモを拡張）:
* v1同様: 2画面、FOV/Near/Farスライダー、箱の外の頂点が赤くなり消える。
* 追加: 「押し潰しアニメーション」ボタン。神の視点で視錐台がグニュッと直方体に正規化され、奥の物体ほど小さくなる様子をスローで見せる（遠近感の発生源の可視化）。
* 追加: 画面内オブジェクト数/クリップされた数のカウンター。
* ガイド付きツアーあり。

### 3.6 ch2d-rasterize.html「ラスタライズ — 三角形がピクセルになる」

**このページで分かること:** ディスプレイはマス目でしか光れない／三角形の内側のマス目を洗い出す作業がラスタライズ／ここまでが「頂点の旅」の終点。

本文構成:

1. **問題提起: ディスプレイは线を引けない** — 画面は小さな光る点（ピクセル）の格子でしかない。斜めの線も、実際はマス目の階段。スマホ画面を拡大した写真の例。
2. **スクリーン空間＝「解像度のマス目座標」** — 1920×1080なら横1920マス・縦1080マス。3D座標がついに「何行目・何列目」という2Dの整数に落ちる。
3. **ラスタライズの手順** — ①三角形の3頂点を画面のマス目位置に置く ②上の行から順にスキャン ③各マスの中心が三角形の内側なら「塗る候補」にする。デモFのスロー再生と完全対応させる。
4. **境界のマスはどうする？** — マスの中心が内側かどうかで判定する、という規則だけ紹介（アンチエイリアスは脚注で名前だけ）。
5. **頂点の旅・全行程のふりかえり** — ローカル→ワールド→ビュー→クリップ→スクリーンの5駅を、駅の路線図ふうの図解1枚にまとめる。各駅で「何のための変換だったか」を一行ずつ復習。この図は以降の章でも再掲する。
6. **よくある誤解** — 「ラスタライズで色が塗られる」→ この段階では「どのマスを塗るべきか」の洗い出しまで。何色にするかは次章（パスの話）。

デモF（v1のラスタライズデモを拡張）:
* v1同様のスロー再生グリッド。
* 追加: 解像度切り替え（16×9 / 32×18 / 64×36）。解像度が上がるほど輪郭が滑らかになるが、塗るマスの数（＝仕事量）が増えることをカウンターで見せる。
* 追加: 三角形の頂点をドラッグで動かせる。
* ガイド付きツアーあり。

### 3.7 ch3a-gbuffer.html「G-Buffer — 絵の『素材』を先に作る」

**このページで分かること:** 現代ゲームは一発で色を塗らない／深度・法線・材質という「素材レイヤー」を先に保存する／なぜ分けるのか。

本文構成:

1. **問題提起: マス目が決まった。すぐ色を塗ればいい？** — 昔のゲームはそうしていた（フォワードレンダリング。用語は軽く）。しかしライトが増えると破綻する。その理由を先に予告してから本題へ。

## 5. 追補（v3）: 数学トラック「文系のための、数式のとびら」

### 5.0 この追補を作る動機

本編（第1〜7章）は「数式を1つも出さない」ことを固く守り、因果関係の理解だけを届けた。
だが読者の中には、本編を読み終えて<strong>「で、その"レシピカード（行列）"の中身は結局どういう数字なの？」「なぜ掛け算で座標が変換できるの？」と、もう一歩踏み込みたくなった熱意ある層</strong>が必ず出てくる。

この追補は、その層――<strong>「数学は学校で苦手だったが、今このテーマになら向き合ってみたい」という文系の再挑戦者</strong>――のためだけに用意する、完全に独立した任意選択トラックである。

重要な認識: <strong>これは本編よりも執筆が難しい</strong>。本編は「数式を消す」ことで易しくできたが、この追補は「数式を出しても逃げられないようにする」必要がある。したがって本追補の主役はコンテンツ（数式）ではなく、<strong>「数式で挫折させないための導線設計」</strong>である。数式を出すこと自体は誰でもできる。文系の人が最後まで読み切れる数式ページを作ることが、このプロジェクト最大の挑戦になる。

### 5.1 対象読者と、絶対に守る約束

* 対象: 本編13ページを読了し、「もっと知りたい」と自ら扉を叩いた<strong>大人</strong>。数学の予備知識は<strong>中学＋高校序盤（文字式・関数のグラフ）で止まっていてよい</strong>。三角比・ベクトル・行列は「聞いたことはあるが使えない」前提。子ども向けに噛み砕くのではなく、大人が忘れているだけの道具を、実物を触りながら思い出させる／使えるようにする、という方針で書く。
* 本追補が読者と結ぶ約束（各ページ冒頭に明示する）:
  1. <strong>この扉は任意である。</strong>読まなくても本編の理解は完結している。「おかわり」であって「宿題」ではない、と最初に安心させる。
  2. <strong>一度に1つの記号しか導入しない。</strong>新しい記号が出たら、その段落ではそれ以外の新記号を出さない。
  3. <strong>すべての式は、必ず本編で触れたデモの動きと1対1で結びつける。</strong>式単独では絶対に提示しない。「さっき動かしたあのスライダーが、この式のこの文字だ」を毎回やる。
  4. <strong>「なぜこの式でうまくいくのか」を、計算ではなく意味で説明する。</strong>導出（証明）は目的にしない。読者が式を"読める"ようになることだけを目指す。
  5. <strong>挫折してよい。</strong>各ページに「ここで難しく感じたら本編に戻ってOK」の離脱ポイントを明記する。罪悪感を持たせない。

### 5.2 教え方の設計原則（このトラックの心臓部）

対象は<strong>大人の再挑戦者</strong>である。子ども向けの言い換え（記号の擬人化・あだ名付けなど）はしない。記号は記号として正面から導入し、そのぶん<strong>「その記号が何を表し、なぜその式で目的が達成できるのか」を、本編と同じインタラクティブな操作で腹落ちさせる</strong>。本編が「数式なしで因果を触らせた」のと同じ方法論を、こんどは数式の上で行う、というのが本トラックの一貫した姿勢である。

全ページ共通で以下の技法を使う。

* <strong>記号は「読み方＋それが指す量」を1行で添えて正面から導入する。</strong>例: 「θ（シータ）＝角度を表す。単位は度。」擬人化やキャラ化はしない。初出時に一度だけ丁寧に、以降は当たり前の道具として使う。
* <strong>「読み下し」を必ず併記する。</strong>すべての数式の直後に、その式を日本語の文章に音読した版を置く。例: 数式の下に「＝『新しいXは、元のXに◯を掛けたものと、元のYに△を掛けたものの和』」と一字一句を言葉にする。式を"読める"ようにすることが本トラックの到達目標。
* <strong>数を代入して手を動かす「電卓タイム」。</strong>抽象式のあと、具体的な数字（例: 角度30°、点(2,0)）を入れて答えが出るまでを実演する。読者にも同じ数字での穴埋めミニ入力欄を出し、正解すると本編デモが連動して同じ結果を描く。
* <strong>インタラクティブな数式ハイライト連動（「生きた数式」ウィジェット）。</strong>数式の各項にマウスを乗せる（タップする）と、対応する図形要素や本編デモのパーツが光る共通部品を実装する。逆に図形を動かすと式中の数値がリアルタイムに更新される双方向連動を基本とする。式と絵が地続きである感覚を作る。
* <strong>「ここまでの1文まとめ」を各節末に置く。</strong>数式を1つ乗り越えるたびに、その式が結局何を言っていたかを数式ゼロの1文に戻す。理解のセーブポイント。
* <strong>インタラクティブ要素の数は各ページで固定しない。</strong>1ページに1つとは限らず、概念の段階（記号の導入・式の組み立て・具体値での確認・本編デモとの接続など）ごとに、理解に必要なだけ複数のウィジェットを配置してよい。式が1つ増えるごとに触れる箇所を1つ用意する、くらいの密度を推奨する。

### 5.3 ページ構成（数学トラック 6ページ + 入口1ページ）

本編とは別ディレクトリ `math/` に置き、本編各章の該当ページ末尾から「▶ もっと踏み込みたい人へ: この裏側の数式を見る（任意）」というリンクで誘導する。数学トラック内にも専用の前後ナビを持たせる。

  m1-negative-grid.html   M1: 負の数と座標平面の再入門（つまずきの原点をほぐす）

各ページは本編と同じ6ブロック型（なぜ→たとえ→仕組み→触って確認→よくある誤解→まとめ）を踏襲しつつ、「触って確認」を<strong>「生きた数式」ウィジェット＋電卓タイム</strong>に置き換える。触って確認のブロックは1ページに複数回登場してよく、式が増えるごとに対応するインタラクティブ要素を足していく。

<strong>この数学トラックには、理解度クイズ・まとめクイズを設けない。</strong>本トラックの目的は正答の確認ではなく「式を自分の手で動かして読めるようになること」であり、その確認はクイズではなくインタラクティブ操作（数式ハイライト連動・電卓タイム・本編デモ連動）そのものが担う。各ページは「まとめ（1文）」で静かに閉じ、次ページ／本編へのリンクに繋ぐ。

#### M0: 数学のとびら（オリエンテーション）
* 目的は<strong>恐怖の除去に全振り</strong>する。数式はほぼ出さない。
* 内容: 「なぜ学校の数学は怖かったのか（＝意味を教わる前に記号を暗記させられたから）」という共感から入り、「このトラックは逆をやる。意味を先に、記号は後で」と宣言。5.1の約束を提示。読む順と、いつでも本編に戻れることを地図で見せる。

#### M1: 負の数と座標平面の再入門
* 本編ch1に対応。多くの文系のつまずきの真の原点である「マイナス」と「座標平面の第2〜4象限」をゆっくりほぐす。
* デモ連動: ch1のデモAを再利用し、式 `x' = x + 3`（平行移動）を「生きた数式」化。スライダーと式の項が同時に光る。

#### M2: 三角比 sin/cos ―「回転」を数にする
* 本編ch2a（Rotation）の裏側。「回転」という操作を数値で表す唯一の道具として sin/cos を導入する。
* たとえ: 観覧車のゴンドラの高さ＝sin、地面の影の位置＝cos。角度θを回すと2つの数が連動して変わる「1つの角度から2つの数が出る装置」として紹介。
* デモ: 単位円ウィジェット（角度スライダー→sin/cosの値がバー表示→点が円周を動く）。回転行列の各成分がここから来ることを予告。

#### M3: ベクトル ― 位置と向きを1セットで持つ
* 本編ch1b/ch3a（法線）の裏側。「(x, y, z)の3つ組」を"1個の矢印"として扱う考え方。足し算＝矢印のしりとり、という絵で。
* デモ: 2本の矢印を足すと第3の矢印になる操作を、本編の頂点デモと連動させる。

#### M4: 行列 ―「レシピカード」の中身を開ける
* 本追補のクライマックス。本編で「中身を見なくていい」と言い続けたレシピカード（行列）を、ついに開ける。
* 核心の見せ方: 2×2行列を「Xの引っ越し先」「Yの引っ越し先」を書いた2枚の指示メモとして提示。行列×ベクトルを「新X＝(元Xの寄与)＋(元Yの寄与)」という<strong>読み下し</strong>で完全に言葉にする。
* デモ: 行列の4つの数字を4つのスライダーで動かすと、単位正方形がリアルタイムに変形（回転・拡大・せん断）する「行列変形ラボ」。回転行列に数値を入れるとM2の単位円と一致することを見せ、点が回る。本編ch2aのワールド行列と接続。

#### M5: 内積 ―「面の向きと光」を1つの掛け算にする
* 本編ch3b（ライティング）の裏側。「法線が光源を向くほど明るい」という言葉の判断が、内積という1つの掛け算に化けることを見せる。
* デモ: 法線ベクトルと光ベクトルの角度を変えると内積の値（＝明るさ）が変わるウィジェットを、本編ch3bの太陽デモと連動。

#### M6: 投影の式 ―「遠くが小さい」を分数で作る
* 本編ch2c（投影）の裏側。遠近感の正体が「奥行きzで割る」という一手であることを見せる。「zで割る」→「zが大きい（遠い）ほど画面上の値が小さくなる」を分数の直感で。
* デモ: ch2cの押し潰しアニメーションに、`画面x = x / z` の式を重ね、zを変えると画面上のxが縮む様子を数式ハイライト連動で示す。

### 5.5 実装方針（追補分）

* 共通部品として <strong>「生きた数式」ウィジェット</strong>を新規実装する: 数式を項ごとの `<span>` に分割し、hover/tapで対応図形をハイライトするデータ属性連携。数式表示はKaTeX等の外部依存を避け、HTML+CSS（上付き・分数はCSSグリッド）で自前描画する（CSPと外部依存排除の方針を継続）。

* 本トラックにはクイズウィジェット（本編共通部品）を配置しない。
