内積 ― 「面の向きと光」を1つの掛け算にする

本編ch3bの照明計算「法線が光源を向いているほど明るい」。あの日本語の判断文が、たった1つの計算に化ける瞬間を見ます。

⏪ このページに必要な道具

  • M3: ベクトル＝矢印。法線は長さ1の単位ベクトル
  • 本編第3章b: 「面が光源を向くほど明るい」という照明の骨格

→ M3を読み直す

1. 問題設定: 「どれくらい同じ方向を向いているか」を数にしたい

照明計算に必要なのは、法線の矢印nと、光源への矢印l（エル。ライトの頭文字）が「どれくらい向きが揃っているか」という1個の数です。完全に同じ向きなら1、直角なら0、真逆なら−1になるような数が欲しい。それがあれば、そのまま明るさに使えます。

この数を作る演算が内積です。

n · l エヌ・ドット・エル（内積）。ベクトル同士の「·」は普通の掛け算ではなく内積という別の演算を表す。結果はベクトルではなく1個の数になるのがポイント。

2. 計算規則は「成分ごとに掛けて、足す」だけ

n · l = nₓ·lₓ ＋ nᵧ·lᵧ

内積は、横成分同士の積と、縦成分同士の積を足したもの。（3Dなら z成分同士の積も足す。項が1個増えるだけ）

規則は簡単ですが、なぜこれが「向きの揃い具合」になるのか。両方が長さ1（M3の単位ベクトル）のとき、内積の値はちょうど2本の矢印の間の角度のcosになります（M2の単位円を思い出してください: cosは「角度0で1、90°で0、180°で−1」と減っていく数でした）。つまり法線を長さ1にそろえておいたのは、内積がそのまま「向きの揃い具合メーター」として読めるようにするための下準備だったのです。M3の伏線がここで回収されます。

下のウィジェットで2本の矢印をドラッグして、①同じ向き→1に近づく ②直角→0 ③逆向き→−1、を確かめてください。

WIDGET 1

向きの揃い具合メーター ― 内積の挙動

3. 照明計算への接続 ― 本編ch3bの正体

あとは読み替えるだけです。nを地面の法線、lを太陽への向きとすれば:

  • 太陽が真上（法線と一致）→ n·l = 1 → 最大の明るさ
  • 太陽が真横（直角）→ n·l = 0 → 光は届かない
  • 太陽が裏側 → n·l がマイナス → 物理的に照らせないので0として扱う（マイナスの明るさは存在しないため切り上げる。この処理を「0でクランプする」と言います）
明るさ = max( 0, n · l )

明るさは、内積の値。ただしマイナスになったら0に切り上げる（maxは「2つのうち大きい方を採る」という関数）。

本編ch3bで太陽スライダーを回したとき、画面の裏では全ピクセルでこの1行が実行されていました。「このマスの表面は上向きで、太陽は右上だから明るめ」という日本語の判断文の正体が、この max(0, n·l) です。下のウィジェットで太陽を回して、地形の各面の明るさが内積の値どおりに変わるのを見てください。

WIDGET 2

ミニ・ライティング ― 各面の明るさ = max(0, n·l)

「GPUは影や陰を"見て"判断している」――GPUは何も見ていません。各マスで n·l という掛け算と足し算を実行しているだけです。それが2百万マス×毎秒60回積み重なると、人間の目には「光が当たっている」ように見える。照明とは、内積の集計結果です。

内積＝成分ごとに掛けて足すと出てくる「向きの揃い具合（cos）」。明るさ = max(0, n·l)。
